Week 5 我们学了**单个 Agent** 如何工作（Text-to-SQL、工作流设计）。

```

  "一个人怎么把任务做好？"
    ↓
Week 6：多 Agent 的协作系统
  "一个团队怎么把任务做好？"
```

5 个核心问题：
1. Agent 怎么记住东西？（记忆管理）
2. 多个 Agent 怎么分工？（协作架构）
3. Agent 之间怎么沟通？（通信机制）
4. Agent 怎么自我改进？（反思机制）
5. 怎么让系统更快？（延迟优化）

## 模块 1：短期记忆 vs 长期记忆

### 为什么记忆很重要？

没有记忆的 Agent 就像**每次对话都从零开始**：
- 不记得用户之前说过什么
- 不记得之前犯过什么错误
- 不记得用户偏好和习惯

有了记忆的 Agent 就像**有经验的助手**：
- 记得你的偏好（喜欢简洁的回答）
- 记得过去的教训（上次 SQL 写错了）
- 越用越懂你

### 短期记忆（对话上下文）

**定义**：当前会话中的信息，即对话历史

**类比**：就像你和朋友打电话时，你能记住这通电话里说过的话

```
用户：帮我查上个月销售额
Agent：销售额是 100 万
用户：那和去年同期比怎么样？  ← Agent 还记得刚才查的是销售额
Agent：同比增长 15%
```

**机制**：填充到 LLM 上下文窗口中的消息

**限制**：
- 受模型上下文窗口大小限制（如 128K tokens）
- 对话太长时，早期的内容会被"挤出去"
- 关掉对话后，短期记忆消失

**处理策略**（当对话太长时）：

1. **滑动窗口**：保留最近 N 条消息，丢弃旧消息
   ```
   优点：简单
   缺点：丢失早期上下文
   ```

2. **摘要压缩**：让 LLM 对旧消息摘要，保留摘要 + 最近消息
   ```
   优点：保留关键信息
   缺点：摘要可能遗漏细节
   ```

3. **Token 预算**：分配比例
   ```
   60% 近期对话（最重要）
   20% 摘要（压缩的旧内容）
   20% 系统指令（Schema、规则等）
   ```

### 长期记忆（跨会话持久化）

**定义**：在对话之间持久化的信息

**类比**：就像你和朋友多次见面后，你记住他的性格、喜好、过去的经历

```
第 1 次对话：
  用户：我喜欢简洁的回答
  Agent 记住了 → 存入长期记忆

第 2 次对话（新的对话）：
  Agent 检索长期记忆 → 发现"用户喜欢简洁回答"
  Agent 给出简洁的回答（而不是冗长的解释）
```

**存储方式**：外部存储（向量数据库、关系型数据库、键值存储）

**三种类型**：

1. **语义记忆**（事实知识）
   ```
   存什么："用户喜欢简洁回答"、"产品A的销售数据"
   怎么存：向量存储（基于语义相似性检索）
   怎么用：根据当前问题，检索相关的知识
   ```

2. **情景记忆**（过去的经历）
   ```
   存什么：过去的交互序列、重要事件
   怎么存：检查点存储（按会话/用户键索引）
   怎么用：回忆上次做了什么，避免重复
   ```

3. **程序记忆**（学到的技能）
   ```
   存什么：学习到的行为和技能、更新的 prompt
   怎么存：配置更新或修正指令
   怎么用：下次遇到类似问题时，用学到的技能
   ```

**记忆的生命周期**：

```
写入（什么时候该记住？）
  ├─ 重要的决策和偏好
  ├─ 用户的纠正和反馈
  ├─ 关键的事实和规则
  └─ 不要记住所有东西！
  └─ 用"重要性评分"过滤：让 LLM 评估是否值得记住

检索（什么时候该回忆？）
  ├─ 新对话开始时 → 检索用户偏好
  ├─ 需要上下文时 → 检索相关事实
  ├─ 遇到类似问题时 → 检索过去的解决方案
  └─ 检索时，相关性 > 句法相似性
  └─ 用嵌入搜索，不是关键词匹配

过期（什么时候该遗忘？）
  ├─ 过时的信息（3 个月前的偏好可能已变）
  ├─ 低重要性评分的记录
  ├─ 冗余或重复的记忆
  └─ 定期清理，避免记忆膨胀
```

**跨 Agent 的记忆共享**：
```
单个用户的记忆：
  namespace: ("users", user_id, "preferences")

多个 Agent 的共享记忆：
  namespace: ("shared", "team", "learnings")

跨会话的记忆：
  持久化到 Postgres/Redis 而不是内存中
```

## 模块 2：多 Agent 协作架构

### 为什么需要多 Agent？

单个 Agent 的局限：
- 只有一种技能（比如只会写 SQL）
- 处理复杂任务时容易出错
- 无法同时处理多个子任务

多个 Agent 的优势：
- 每个 Agent 专注一种技能（专精）
- 可以分工协作（效率）
- 互相验证和补充（准确性）

### 三种基本架构

#### 顺序架构（流水线）

```
Agent A → Agent B → Agent C

就像工厂流水线：
  研究员 → 撰写员 → 审核员 → 编辑员
  每一步的输出是下一步的输入
```

**适用场景**：任务有清晰的分阶段步骤
**优势**：简单、可预测、易调试
**劣势**：
- 慢（总延迟 = 所有 Agent 延迟之和）
- 错误会级联（A 出错 → B 也出错 → C 也出错）

**工程提示**：
- 在每个 Agent 之间添加验证门控
- 如果 B 的输入质量低，提前失败而不是传播错误

#### 并行架构（并发）

```
       ┌→ Agent A（查文档1）→┐
用户 → ├→ Agent B（查文档2）→├→ 合并结果 → 返回
       └→ Agent C（查文档3）→┘

就像多人同时做不同的事：
  3 个人同时分析 3 个文档
  结果汇总后返回
```

**适用场景**：任务可分解为独立子任务
**优势**：快（延迟 = 最大子任务延迟，不是总和）
**劣势**：
- 合并结果需要协调
- 可能产生冲突的输出
- 更难调试（哪个 Agent 出了问题？）

**合并策略**：
- 投票：多个 Agent 投票决定最终答案
- 加权平均：根据置信度加权
- LLM 仲裁：让另一个 Agent 做裁判
- 结构化组合：A 负责研究，B 负责分析，C 合并

#### 分层架构（Supervisor-Worker）

```
Supervisor（总指挥）
  ├→ Worker A（SQL 专家）→ 返回 SQL 结果
  ├→ Worker B（搜索专家）→ 返回搜索结果
  └→ Worker C（计算专家）→ 返回计算结果
  ↓
Supervisor 整合所有结果 → 返回最终答案

就像公司架构：
  老板分配任务给不同部门
  各部门完成后汇报
  老板综合决策
```

**适用场景**：复杂任务需要动态分解和协调
**优势**：
- 灵活（Supervisor 可根据结果调整计划）
- 自然错误处理（Worker 返回不好时，Supervisor 可重新分配）
- 可扩展（添加更多 Worker）

**劣势**：
- Supervisor LLM 调用增加开销
- 如果 Supervisor 的分解不好，整个流程都受损

**变体**：
```
嵌套分层：
  Top Supervisor → Research Supervisor → 3 Researchers

Supervisor + 工具：
  Supervisor 不仅调用 Agent，也可以直接调用工具
  （混合架构，更灵活）
```

### 选择决策树

```
问：任务是什么性质？

线性分阶段？（步骤1 → 步骤2 → 步骤3）
  → 顺序架构

子任务独立？（可以同时做）
  → 并行架构

复杂不确定？（需要动态分配和调整）
  → 分层架构

混合依赖？（部分独立 + 部分有依赖）
  → 分层 + 并行子步骤
```

### 和 Week 5 的联系

```
Week 5 的混合策略 → Week 6 的架构选择
  都是"根据场景选最适合的方案"

Week 5 的复杂度分类 → Week 6 的任务路由
  简单任务 → 单 Agent
  复杂任务 → 多 Agent 协作
```

## 模块 3：Agent 通信机制

### 为什么通信机制很重要？

多个 Agent 协作时，需要**交换信息**：
- Agent A 的研究结果怎么传给 Agent B？
- 多个 Agent 的结果怎么合并？
- 出错了怎么通知其他 Agent？

### 三种传递方式

#### 共享状态（黑板模式）

```
想象一个公共黑板：
  所有 Agent 都能看到黑板上的内容
  Agent A 把研究结果写到黑板上
  Agent B 从黑板上读取研究结果
  Agent C 把审核意见写到黑板上
```

**优势**：
- 所有 Agent 对当前情况有一致的视图
- 易添加新 Agent（只需读写黑板）

**劣势**：
- 状态污染（一个 Agent 的输出可能干扰另一个）
- 需要仔细的命名约定和类型定义

**LangGraph 的实现**：
- Agent 在共享 `State` 对象上运行
- 每个节点读取和更新状态字段
- 例如：`research_results: list[str]`，`draft: str`，`review_comments: list[str]`

#### 直接消息传递

```
就像发邮件：
  Agent A 直接给 Agent B 发消息
  Agent B 收到后处理
  其他 Agent 不知道这封信的内容
```

**优势**：
- 隐私（Agent 之间不会看到彼此的所有数据）
- 明确的沟通意图

**劣势**：
- 需要显式路由逻辑（谁发给谁？）
- Agent 可能错过相关信息

#### 事件驱动（发布-订阅）

```
就像微信群的通知：
  Agent A 发布事件："研究完成了！"
  所有订阅了这个事件的 Agent 都会收到通知
  Agent B 和 Agent C 分别响应
```

**优势**：高度解耦（Agent 不需要知道彼此）
**劣势**：难以追踪因果链；事件顺序不可预测

### 通信内容的格式

#### 结构化消息（推荐）

```python
# 不好的做法（自由文本）
Agent A → Agent B: "我发现用户喜欢简洁的回答，置信度大概80%"

# 好的做法（结构化消息）
class ResearchResult:
    title: str           # "用户偏好分析"
    findings: list[str]  # ["喜欢简洁回答", "偏好中文"]
    confidence: float    # 0.8
    source: str          # "Agent_A"
    timestamp: datetime  # 2026-04-29 15:30
```

**优势**：下游 Agent 可程序化解析和验证，不需要让 LLM 解释自由文本

#### 混合方法（最佳实践）

```
关键管道：用结构化数据
  - SQL 查询、工具调用、验证结果
  - 确保 Agent 之间传递的数据可解析

上下文/推理：用自然语言
  - 为什么做出某个决策
  - 对结果的评价和解释
  - 给下游 Agent 提供思考背景
```

### 通信最佳实践（4 条）

1. **定义清晰的接口**
   - 每个 Agent 有定义的输入 schema 和输出 schema
   - 像微服务 API 合约一样

2. **添加元数据**
   - 来源 Agent、置信度、时间戳
   - 下游 Agent 可以评估信息可靠性

3. **保留版本**
   - draft_v1, draft_v2
   - Reviewer 可以看到变更轨迹

4. **定义失败模式**
   - 超时怎么办？
   - 输出格式错误怎么办？
   - 至少：错误消息 + 默认/回退行为

## 模块 4：Self-critique 反思机制

### 为什么需要反思？

LLM 会犯错，但它**不知道自己错了**：
- 幻觉（自信地说出错误的事实）
- 遗漏（忘了重要的业务规则）
- 逻辑错误（推理过程有 bug）

反思机制让 Agent **检查自己的输出**，发现并修复错误。

### 三种反思模式

#### 1. 自我反思循环

```
Agent 生成输出
  ↓
Agent 评估自己的输出
  ↓
发现缺陷 → 修订
  ↓
再次评估
  ↓
满足标准 → 输出最终结果
```

**实现**：两次 LLM 调用
- 第 1 次：生成（"请生成 SQL"）
- 第 2 次：反思（"请检查这个 SQL 是否有错误"）

**适用**：质量关键且外部验证昂贵时

#### 2. Reflexion 模式（Shinn et al., 2023）

```
核心创新：反思记忆

不只是当前检查自己的输出
而是把每次的错误教训**长期记住**

流程：
  执行 → 观察 → 生成反思 → 存储反思 → 下次检索指导行为

示例：
  第 1 次尝试：
    生成 SQL → 执行 → Error "column not found"
    反思："我以为 user_name 存在，实际是 username。下次先检查 Schema"

  第 2 次尝试：
    检索反思记忆 → 发现"先检查 Schema"
    生成 SQL → 先检查字段 → 成功！

关键：反思作为长期记忆，过去的错误教训指导未来尝试
```

**与 Week 5 的联系**：
- 这就是 Day 3 学的"反馈循环"的高级版本
- 不仅收集失败案例，还把教训**存储为记忆**

#### 3. 多角度批评

```
不是简单问"这个输出好不好"
而是从多个角度评估：

角度 1：准确性 — 事实是否正确？
角度 2：完整性 — 是否遗漏了重要方面？
角度 3：清晰度 — 用户是否能理解？
角度 4：安全性 — 是否有有害内容或隐私泄露？

每个角度可以是独立的 LLM 调用
也可以是一次调用，但有详细的 checklist
```

**优势**：比单一"评估好坏"更全面

### 工程实现要点

**Prompt 设计**：
```
❌ 通用反思（容易模糊）
  "请评估这个 SQL 是否正确"
  → LLM 可能回复"看起来还行"

✅ 具体 checklist（精准）
  "请检查这 5 个错误类别：
   (1) 字段是否存在
   (2) 是否遗漏 cancelled 过滤
   (3) JOIN 是否正确
   (4) 聚合逻辑是否有误
   (5) 是否存在 SQL 注入风险"
  → LLM 会逐一检查，给出具体反馈
```

**迭代限制**：
- 无限制反思可能无限运行（Agent 总觉得"可以改进"）
- 2-3 次是常见最佳实践
- 设置收敛条件：修订的变更低于阈值时停止

**成本控制**：
- 每次反思使 LLM 调用成本翻倍（生成 + 反思 = 2 次调用）
- 只在复杂/高风险任务上应用反思
- 小模型反思 + 大模型生成（反思本身更容易——是在 checklist 上模式匹配）

**反思的局限**：
```
❌ LLM 无法检测自身幻觉（自信地认为错误事实是正确的）
❌ LLM 可能"过度修正"（修复小问题时引入新错误）

补救：
  结合自我反思 + 外部验证（工具执行、数据库查询、事实核查）
  → 这就是 Week 5 的验证层级和反思的结合
```

## 模块 5：延迟优化和并行调用

### 为什么延迟优化很重要？

用户等待时间直接影响体验：
- 1-2 秒：即时响应，用户满意
- 3-5 秒：可以接受，但有点慢
- 10+ 秒：用户会焦虑，可能放弃
- 30+ 秒：用户大概率离开

### 5 种优化策略

#### 1. 并行工具调用

```
顺序执行（慢）：
  get_weather("London")    2s
  get_weather("Tokyo")     2s
  get_exchange_rate()      2s
  总延迟：6s

并行执行（快）：
  get_weather("London")    ┐
  get_weather("Tokyo")      ├→ 同时执行
  get_exchange_rate()       ┘
  总延迟：2s（最大的一个）

Anthropic Claude API 直接支持：
  单个 API 响应可包含多个 tool_use
  客户端并行执行所有工具调用
  将所有结果一起返回
```

**前提条件**：工具调用之间**不相互依赖**
- 如果 Tool B 需要 Tool A 的输出 → 必须顺序执行
- 如果 Tool A 和 Tool B 独立 → 可以并行执行

#### 2. 流式输出（三层）

```
第 1 层：Token 流式
  生成 token 时就流式传输文本
  用户看到文字逐渐出现（而不是等全部生成完）
  所有主流 LLM API 都支持

第 2 层：事件流式
  流式传输结构化事件
  "正在搜索..." → "找到 3 个来源" → "正在生成答案..."
  让用户知道系统在做什么，减少焦虑

第 3 层：部分结果流式
  验证完成前就流式传输中间输出
  比如草稿先展示，审核后再更新
```

#### 3. 缓存策略

```
语义缓存：
  存储过去的 (query, response) 对
  语义相似的查询 → 返回缓存响应
  重复查询延迟 ↓ 10x

工具结果缓存：
  天气数据缓存 30 分钟
  数据库 Schema 缓存 1 小时
  减少重复的工具调用

Prompt 缓存（Week 5 已学）：
  稳定前缀的 prompt 降低成本/延迟
  命中缓存率：Prompt 中有多少内容命中了之前缓存的内容
  命中率越高 → 成本越低 → 响应越快
```

#### 4. 复杂度路由

```
这就是 Week 5 的分层策略！
但视角从成本 → 转向延迟：

简单 → 单次 LLM 调用，1-2s
中等 → LLM + 1-2 工具，3-5s
复杂 → LLM + 多工具 + 反思，10-30s

关键：不是所有问题都需要完整流程
```

#### 5. 延迟预算框架

```
定义延迟预算：
  简单查询 < 5s
  中等查询 < 15s
  复杂查询 < 30s

为每个步骤分配子预算：
  LLM 推理 < 3s
  工具调用 < 2s
  反思检查 < 3s

超预算时触发降级：
  跳过反思 → 使用缓存 → 返回部分答案
```

### 并行调用的最佳实践（6 条）

1. **验证独立性**：确认工具调用确实不相互依赖
2. **限制并发**：最多 5-10 个并行调用，避免资源耗尽
3. **独立超时**：每个调用有自己的超时，不让一个慢的工具阻塞整个批次
4. **优雅降级**：部分失败返回部分结果 + 错误通知
5. **单独重试**：只重试失败的调用，不重试整个批次
6. **记录耗时**：追踪每个工具调用的时间，识别瓶颈
