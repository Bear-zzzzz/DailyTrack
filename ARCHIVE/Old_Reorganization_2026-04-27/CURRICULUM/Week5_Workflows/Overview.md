## 🎓 核心概念

### Task Breakdown（任务拆分）

**为什么要拆分**
1. 降低单个 task 的复杂度
2. 支持并行执行
3. 便于错误恢复
4. 提高整体可靠性

**两种模式**
- **显式规划**：Agent 自己规划任务
- **预定义工作流**：人工设定固定流程

### Text-to-SQL 难点深度分析

**难点 1：Schema 理解**
- 问题：表字段众多、命名不统一、关系复杂
- 解决：完整的 Schema 文档 + 统一命名规范
- 效果：准确率 ↑ 30-40%

**难点 2：语义转换**
- 问题：自然语言歧义、业务逻辑复杂
- 解决：Query 改写 + Few-shot 示例
- 效果：准确率 ↑ 20-30%

**难点 3：复杂逻辑**
- 问题：CTE、窗口函数、子查询
- 解决：递进式 Few-shot 示例
- 效果：准确率 ↑ 40-60%（复杂查询）

### Few-shot 的心理学效应

```
前面简单例子 → LLM 预热，建立基本规律
中间例子 → 建立信心，扩展理解
复杂例子 → 说明边界情况，展示能力

结果：准确率 ↑ 15-25%（vs Zero-shot）
```

## 🛠️ 技术细节

### SQL 验证的 4 层级

| 层级 | 检查内容 | 成本 | 有效性 |
|------|---------|------|--------|
| 1 | 语法正确 | 🟢 低 | 🟡 中 |
| 2 | Schema 存在 | 🟢 低 | 🟡 中 |
| 3 | 实际执行 | 🔴 高 | 🟢 高 |
| 4 | 业务规则 | 🟢 低 | 🟢 高 |

**建议**：Level 1-2 + Level 4 必做，Level 3 按需做

### 成本优化三层策略

1. **简单问题**（70%）：用模板 → 成本 ↓ 95%
2. **中等问题**（25%）：用缓存 LLM → 成本 ↓ 80%
3. **复杂问题**（5%）：用缓存 + 验证 → 成本 ↓ 30%
