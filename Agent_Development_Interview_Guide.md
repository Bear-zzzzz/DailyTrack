## 第一层：LLM 基础理论（必须掌握）

### 模型训练和架构

#### **Transformer 架构**
- **Encoder-Decoder 区别**
  - Encoder：编码输入（BERT 类模型）
  - Decoder：生成输出（GPT 类模型）
  - 选择：生成任务用 Decoder-only（更快），理解任务可用 Encoder

- **Self-Attention 机制**
  - 通过 Q、K、V 矩阵计算相关性
  - 允许模型"看到"整个序列的任何位置

- **Multi-head Attention**
  - 多个注意力头并行工作
  - 每个头学不同的相关性模式

- **位置编码**
  - 为什么需要？Attention 本身不感知位置信息
  - 绝对位置 vs 相对位置编码

#### **大模型训练流程**
```
预训练（Pre-training）
  ↓ 大规模无标签数据，Next Token Prediction
指令微调（Instruction Tuning）
  ↓ 有标签数据，教会模型遵循指令
强化学习反馈（RLHF）
  ↓ 人工反馈，对齐人类价值观
  ↓
生产模型
```

- **Token Prediction 本质**：给定前 N 个 Token，预测第 N+1 个
- **为什么 LLM 产生幻觉**
  - 参数知识和检索信息冲突
  - 训练数据中的噪声
  - 模型过度自信

#### **Function Call 机制（重点！）**

**什么是 Function Call？**
- 模型学会输出结构化的工具调用指令
- 本质：在指令微调阶段加入工具调用示例

**工作流程**
```
用户输入
  → 模型判断需要调用什么工具
  → 输出 JSON/XML 格式的函数调用
  → 系统执行函数
  → 结果返回给模型
  → 模型生成最终回答
```

**关键细节**
- 工具定义怎么写？
  - Schema 描述（函数名、参数类型、约束）
  - 自然语言描述
  - 使用示例

- 模型怎么学会调用？
  - In-context Learning：通过示例提示（Few-shot）
  - 微调：在指令微调数据中加入工具调用示例

- 调用策略
  - 一次调用：一个 Agent Turn 只调一个工具
  - 并行调用：同时调多个工具
  - 顺序调用：前一个结果作为后一个输入

- 错误处理
  - 工具返回错误 → 模型是否会自动重试？
  - 需要设计容错机制

**Function Call vs MCP 的区别**
- Function Call：模型输出函数调用信息，系统执行
- MCP（Model Context Protocol）：
  - Anthropic 提出的标准协议
  - 规范化 LLM 和工具的交互
  - 包含权限控制、错误处理、标准化工具描述

#### **模型微调方案**

| 方案 | 参数量 | 成本 | 效果 | 适用场景 |
|------|--------|------|------|---------|
| **LoRA** | 原模型的 0.1-1% | 低 | 中等 | 任务特定优化、资源受限 |
| **QLoRA** | 同 LoRA + 4-bit 量化 | 最低 | 中等 | 显存极其受限 |
| **Full Finetune** | 100% | 高 | 最好 | 数据充分、资源充足 |
| **Adapter** | 1-5% | 低 | 中等 | 多任务学习 |

**LoRA 原理**
- 冻结原模型，只训练小的 A、B 矩阵
- $W' = W + AB^T$（秩为 r，通常 8-64）
- 优点：参数少、训练快、显存少、推理无额外延迟

**指令微调 vs 领域微调**
- 指令微调：教会模型遵循指令格式（基础）
- 领域微调：在特定领域数据上持续学习（深化）
- 最佳实践：预训练 → 指令微调 → 领域微调

**高频题**
- 你们怎么选择微调方案的？
- 成本和效果如何权衡？
- 数据量多少时值得微调？

#### **Embedding 和分词**

**Embedding 是什么？**
- 将文本转为向量（通常 768-1536 维）
- 语义接近的文本，向量距离也接近
- 是 RAG 系统的基础

**常用 Embedding 模型**
- OpenAI text-embedding-3-small/large
- 智谱 Embedding
- 开源：bge-large-zh、m3e、jina-embeddings
- 选择标准：维度、语言支持、性能、成本

**分词器（Tokenizer）**
- 为什么分词？
  - Token 是计费的基础
  - 模型实际处理的是 Token 而非字符

- BPE（Byte Pair Encoding）原理
  - 从单个字节开始，迭代合并频繁出现的字节对
  - 结果是变长编码

- 中文 vs 英文分词差异
  - 英文：一个单词通常 1-2 Token
  - 中文：一个字通常 1-2 Token，一个词 2-4 Token
  - 中文文本通常需要更多 Token

## 第二层：Agent 框架和工具（实战层）

### 主流框架对比

| 框架 | 特点 | 适用场景 | 高频题 |
|------|------|---------|--------|
| **LangChain** | 功能全面、易上手、社区大 | 快速原型、教学、中小型项目 | Chain 和 Agent 的区别？如何组织复杂流程？ |
| **LlamaIndex** | 专注 RAG、数据连接器丰富 | 文档 QA、知识库检索 | 如何选择 Retriever？ |
| **AutoGen** | 多 Agent 协作、支持对话 | 需要角色划分的复杂任务 | 如何设计 Agent 间的通信？ |
| **Crew AI** | 工作流和 Task 的分离 | Agent 编排、工作流管理 | Task 和 Role 的设计如何分离？ |
| **自研** | 完全掌控、性能优化 | 性能要求高、特定需求 | 为什么手搓而不用框架？ |

**选择建议**
- 快速开发：LangChain
- 专注 RAG：LlamaIndex
- 需要多 Agent：AutoGen 或 Crew AI
- 高性能、特殊需求：自研

### LangChain

**核心概念**
- **Chain**：一系列操作的顺序组合（确定性流程）
- **Agent**：动态选择工具的流程（需要决策）
- **Tool**：Agent 能调用的函数
- **Retriever**：从文档库中检索的模块

**什么时候用 Chain，什么时候用 Agent？**
- Chain：流程固定、步骤明确（如 A→B→C）
- Agent：需要动态选择（如 ReAct 问题解决）

**如何组织复杂流程？**
- 多个 Chain 串联
- 条件分支（if-then）
- 循环和重试

### LlamaIndex

**核心价值**：RAG 的完整工具链
- 加载数据（Document Loaders）
- 分割文本（Node Parsers）
- 索引创建（Index Creators）
- 查询引擎（Query Engines）

**Retriever 选择**
- Vector Retriever：语义相似度
- Keyword Retriever：BM25
- Hybrid Retriever：混合
- 上下文检索：自动扩展上下文

### AutoGen

**多 Agent 协作**
- ConversableAgent：可进行对话的 Agent
- 通过消息传递通信
- 支持多轮对话和群组讨论

**如何设计 Agent 间的通信？**
- 显式消息：A 发消息给 B
- 隐式消息：通过共享状态
- 投票机制：多个 Agent 投票

### MCP（Model Context Protocol）

**是什么？**
- Anthropic 提出的标准协议
- 规范化 LLM 和工具的交互
- 包含权限、安全、标准化

**vs Function Call**
- Function Call：模型输出工具调用信息
- MCP：完整的生态和协议标准
  - 工具发现机制
  - 权限管理
  - 错误处理标准
  - 认证机制

**何时用？**
- 需要集成多个工具源
- 对安全和标准化有要求
- 企业级应用

**有没有实践过？**
- 与 Claude Code 的集成
- 搭建本地工具服务

### LiteLLM（网关框架）

**核心价值**
- 统一多个 LLM 供应商的 API 接口
- 单一代码库支持 100+ 模型

**使用场景**
- **多模型切换**：根据成本/性能动态选择
- **容错**：主模型失败，自动切换备用
- **成本优化**：便宜模型处理简单任务，复杂任务用贵模型
- **统一日志和监控**：所有 LLM 调用的可观测性
- **请求重试**：内置重试机制
- **速率限制**：防止超额使用

**集成示例**
```python
from litellm import completion

# 自动处理 API 密钥、格式转换、错误处理
response = completion(
    model="gpt-4",  # 可切换到其他模型
    messages=[{"role": "user", "content": "Hello"}],
    fallbacks=["gpt-3.5-turbo", "claude-3-sonnet"]  # 失败后的备用
)
```

### 为什么手搓 Agent 而不是用框架？

**框架的限制**
- 性能：框架可能有额外开销
- 定制化：特殊需求可能需要修改框架代码
- 学习成本：理解框架的设计哲学

**手搓的优势**
- 完全掌控：每一行代码都清楚
- 性能优化：针对性优化
- 简洁性：只实现需要的功能

**权衡**
- 快速原型：用框架
- 生产系统：混合（框架 + 定制化）

**高频题**：你为什么手搓 Agent 而不用框架？
- 预期答案：性能要求 / 特定的工作流需求 / 成本控制

## 第三层：Prompt 工程（高频题区）

### 推理范式对比

| 范式 | 原理 | 何时用 | 缺点 | Token 增加 |
|------|------|--------|------|-----------|
| **直接回答** | 一步到位 | 简单问题 | 复杂问题错误率高 | 无 |
| **CoT（思维链）** | 模型逐步推理，输出思考过程 | 数学、逻辑题 | 不一定更准确，增加延迟 | 2-3 倍 |
| **ReAct** | Reasoning + Acting，在思考中调用工具 | Agent 的标准方式 | 需要工具支持 | 中等 |
| **Tree-of-Thought** | 多条推理路径并行探索 | 需要深层搜索的问题 | 计算成本高 | 5-10 倍 |
| **自洽性** | 多次推理取多数答案 | 提高准确率 | 成本 3-5 倍 | 3-5 倍 |

### CoT（Chain-of-Thought）

**为什么 CoT 有效？**
- 模型输出中间推理步骤
- 减少推理错误的可能
- 提高复杂问题的准确率
- 人类可以验证每一步

**什么时候不应该用 CoT？**
- 简单问题（直接回答更快）
- 需要低延迟的场景
- Token 成本敏感
- 模型本身很强大

**CoT 的变体**
- **Zero-shot CoT**：直接提示"让我们一步步思考"
- **Few-shot CoT**：提供示例
- **Self-consistency**：多次生成，投票选择

### ReAct（Reasoning + Acting）

**核心概念**
```
思考（Thought）→ 动作（Action）→ 观察（Observation）→ 重复
```

**工作流程**
1. 模型思考需要做什么
2. 模型选择一个工具并调用
3. 系统执行工具并返回结果
4. 模型观察结果，继续思考
5. 重复直到完成

**为什么有效？**
- 结合了推理（思考）和执行（工具调用）
- 动作提供的反馈帮助模型纠正
- 天然支持 Agent 架构

**vs CoT**
- CoT：只有思考，没有实际行动
- ReAct：思考 + 执行 + 反馈循环

### Prompt Caching

**作用**
- 缓存重复的长上下文
- 节省 Token 和成本
- 提高响应速度

**实现方式**
- OpenAI：cache_control 参数
- 标记需要缓存的部分（通常是系统提示、长文档）
- 缓存在服务端，多个请求复用

**成本效益**
- 缓存命中率高（如 90%）
- 缓存内容成本：原价的 10%
- 创建缓存成本：原价
- 需要重复请求才能收益

**何时用？**
- 长系统提示（>1000 Token）
- 固定的上下文文档
- 多个用户/请求共享资料

### 采样参数调优

#### **Temperature（温度）**
- **范围**：0-2（通常 0-1）
- **作用**：控制随机性
  - 0：确定性，总选最高概率
  - 1：平衡
  - >1：更随机

- **建议设置**
  - 精确任务（翻译、分类、数据提取）：0-0.3
  - 平衡任务（总结、问答）：0.5-0.7
  - 创意任务（创意写作、头脑风暴）：0.7-1.0

#### **Top-p（核采样）**
- **范围**：0-1
- **作用**：累积概率阈值
  - 只考虑累积概率达到 p 的候选词
  - 防止选太小概率的词

- **建议设置**
  - 通常 0.9（推荐）
  - 与 Temperature 配合使用
  - 单独用 Temperature 通常就够了

#### **Top-k**
- **范围**：整数
- **作用**：只从概率最高的 k 个词中采样
- **建议**：很少单独用，和 top-p 作用重复

#### **Frequency Penalty（频率惩罚）**
- **范围**：-2 到 2
- **作用**：惩罚已出现过的词，鼓励多样性
- **建议设置**
  - 避免重复：0.5-1.0
  - 要求多样性的任务（创意、列表）：1.0-2.0

#### **Presence Penalty（出现惩罚）**
- **范围**：-2 到 2
- **作用**：鼓励引入新话题
- **建议设置**
  - 通常用于创意任务：0.5-1.0

#### **最佳实践**
```
精确任务：temperature=0.3, top_p=1.0
常规任务：temperature=0.7, top_p=0.9
创意任务：temperature=0.9, top_p=0.95, frequency_penalty=1.0
```

## 第四层：RAG 系统（系统设计题高频）

### RAG 核心链路

```
用户查询
  ↓
1. Query Rewrite（查询改写）
  ↓
2. Multi-path Retrieval（多路召回）
  ├─ 向量检索
  ├─ BM25 关键词
  └─ 其他数据源
  ↓
3. Ranking/Reranking（排序）
  ↓
4. Context Assembly（上下文组织）
  ↓
5. Generation（生成答案）
  ↓
最终答案（含引用）
```

### 文档切割策略

**问题**
- 太小（512 Token）：丢失上下文，检索碎片化
- 太大（4000 Token）：检索精度下降，包含噪音

**常见方案**

1. **固定大小切割**
   - 大小：512 Token + 50 Token 重叠
   - 优点：简单、均匀
   - 缺点：可能切割关键信息

2. **语义感知切割**
   - 按句子、段落、章节切割
   - 优点：保留语义
   - 缺点：大小不均

3. **树形切割**
   - 保留文档的层级结构
   - 小片段 → 大片段 → 完整文档
   - 优点：支持多粒度检索
   - 缺点：复杂度高

**怎么规避语义被切割掉？**
- 结构化分割：按文档结构（标题、段落）
- 保留关键短语：识别关键词并确保完整
- 上下文补全：添加前后段落的摘要
- 元数据附加：添加标题、标签等

### 多路召回

**含义**：不只用一种检索方式

**为什么多路？**
- 提高覆盖率（某种方式可能遗漏）
- 处理多种查询类型
- 结合各方式优势

**常见方案**

1. **向量检索 + BM25**
   - 向量：语义相似度
   - BM25：关键词匹配
   - 组合：融合得分

2. **向量检索 + 图数据库**
   - 向量：相似性
   - 图：关系推断（多跳）
   - 场景：需要推导关系

3. **向量检索 + 元数据过滤**
   - 先用元数据（时间、类别）过滤
   - 再在子集中向量检索
   - 场景：多租户、大规模数据

4. **多个 Embedding 模型**
   - 用不同的 Embedding 模型
   - 投票或融合结果
   - 场景：需要高精度

**融合方法**
- 加权求和：`score = w1 * vector_score + w2 * bm25_score`
- 排序融合（RRF）：基于排序位置融合
- 学习融合：训练模型学习最优融合权重

### 向量数据库选择与性能

| 数据库 | 量级 | 性能 | 特点 | 成本 | 适用场景 |
|--------|------|------|------|------|---------|
| **Pinecone** | 10B+ | 高 | 云托管、开箱即用 | 中等（按量计费） | 无需自建、需要托管 |
| **Qdrant** | 1B+ | 很高 | 开源、高性能、量化支持 | 低（自建） | 高性能、中等规模 |
| **Milvus** | 10B+ | 高 | 开源、分布式、社区活跃 | 低（自建） | 超大规模、分布式 |
| **Weaviate** | 1B+ | 中等 | 支持混合搜索、GraphQL API | 低（自建） | 需要混合查询 |
| **Chroma** | 100M | 低 | 轻量级、本地、易上手 | 很低 | 本地开发、小数据 |

**性能指标**
- **吞吐量**：每秒查询数（QPS）
- **延迟**：单次查询时间（毫秒）
- **准确率**：Recall@k、NDCG@k

**高频题**
- 你们的数据量是多少？有没有遇到性能瓶颈？
  - 预期答案：具体数字 + 遇到的问题 + 解决方案

**Qdrant 性能特点**
- 高性能：HNSW 算法（层次可导航小世界图）
- 量化：减少显存占用
- 滤波：高效的元数据过滤
- 分布式：支持集群部署

### 图数据库用途

**为什么要用？**
- 关系推断：A 认识 B，B 认识 C → C 与用户可能相关
- 多跳推理：通过多个关系链找到答案
- 实体消歧：同名但不同含义的实体

**常见场景**
- 知识图谱：存储实体和关系
- 推荐系统：用户-物品-类别图
- 复杂查询：需要多跳遍历

**与向量库的区别**
- 向量库：相似度搜索
- 图库：关系遍历
- 组合：先用图找到相关实体，再用向量精排

### 规避幻觉

**根本原因**
- 模型的参数知识过时
- 检索文档不相关
- 模型对不确定信息过度自信

**规避方案**

1. **检索精度优化**
   - 多路召回提高覆盖率
   - Rerank 改进排序
   - Query 改写理解意图

2. **强制引用机制**
   - 生成答案时必须标注来源
   - 格式：答案 + [引用文档 ID]
   - 人工可验证

3. **验证机制**
   - 生成答案后，用检索验证答案内容
   - 答案中的关键事实是否出现在检索结果中
   - 不符合则拒绝或重新生成

4. **置信度评估**
   - 模型评估自己的答案可信度
   - 低置信度时拒绝或转人工

5. **黑名单机制**
   - 识别高风险问题（健康、法律建议）
   - 拒绝直接回答或转人工

### 微调 vs RAG 对比

| 方面 | 微调 | RAG | 组合方案 |
|------|------|-----|---------|
| **成本** | 高（训练计算） | 中等（存储+检索） | 中高 |
| **知识更新** | 慢（需重新训练） | 快（直接更新文档） | 最优 |
| **准确性** | 高（深层融合） | 中（可能检索错误） | 最高 |
| **学习能力** | 强（模型学习） | 弱（只是检索） | 平衡 |
| **何时用** | 长期性格改变 | 知识库型问答 | **推荐方案** |

**微调 + RAG 的最优实践**
1. 用 RAG 做基础问答
2. 针对常见/重要问题微调
3. 微调后的模型 + RAG 形成闭环
4. 不断收集失败案例，迭代改进

### 效果量化（关键！面试必问）

**检索效果评估**

1. **Recall@k（召回率@k）**
   - 定义：相关文档中有多少被召回的前 k 个结果中
   - 公式：`Recall@k = 相关文档数 ∩ 前k结果 / 总相关文档数`
   - 目标：Recall@10 > 0.8

2. **Precision@k（精准率@k）**
   - 定义：前 k 个结果中有多少是相关的
   - 公式：`Precision@k = 相关文档数 ∩ 前k结果 / k`
   - 目标：Precision@1 > 0.9

3. **NDCG@k（归一化折扣累计增益）**
   - 考虑排序质量
   - 相关度高的结果排前面时评分高
   - 目标：NDCG@10 > 0.7

4. **MRR（平均倒数排名）**
   - 第一个相关结果的平均位置倒数
   - 公式：`MRR = 1/排名`
   - 适合找"最佳答案"的场景

**回答效果评估**

1. **自动化指标**
   - BLEU/ROUGE：与参考答案的文本相似度
   - F1 Score：精准率和召回率的平衡
   - 信息覆盖率：答案是否涵盖所有关键信息

2. **人工评分标准**
   - 准确性（Correctness）：0-3 分
   - 相关性（Relevance）：0-3 分
   - 完整性（Completeness）：0-3 分
   - 清晰度（Clarity）：0-3 分

3. **业务指标**
   - 用户满意度（点赞/点踩）
   - 后续提问率（用户是否需要补充问题）
   - 任务完成率

4. **引用准确率**
   - 模型引用的文档是否确实包含相关信息
   - 百分比：准确引用数 / 总引用数

**监控看板建议**
```
实时指标：
  - 平均查询延迟
  - 缓存命中率
  - 错误率

日度指标：
  - Recall@10（检索）
  - NDCG@10（排序）
  - 用户满意度（人工标注）

周度指标：
  - 各类问题的准确率分布
  - 常见失败案例分析
  - 模型和检索的贡献度
```

## 第五层：工作流设计（高频题）

### 任务拆分与规划

**为什么要拆分？**
- **准确率提高**：一个大任务容易出错，多个小任务更可靠
- **并行化**：独立子任务可并行执行，降低延迟
- **可调试性**：每个子任务的输出可验证
- **容错性**：某个子任务失败可独立处理

**怎么拆分？**

1. **显式规划**
   - Agent 看到任务，自己分解成子任务
   - ReAct 框架：先思考拆分方案
   - 优点：灵活，可应对不同任务
   - 缺点：可能拆分不好

2. **预定义工作流**
   - 人工设计固定的任务 DAG
   - 优点：稳定、可控
   - 缺点：不灵活

3. **混合方案**
   - 有一些固定流程，Agent 在必要时灵活调整
   - 推荐方案

**效果评估**
- 完成率：多少百分比的任务被成功完成
- 成功率：成功完成的任务中多少满足质量要求
- 端到端延迟：从输入到最终答案的时间
- 单步准确率：每个子任务的准确率

### Text-to-SQL（高频系统设计题）

**难点**
- 理解查询意图：用户的自然语言描述可能模糊
- 正确映射：字段名、表名的对应
- 复杂 JOIN：多表关联的语法和逻辑
- 业务逻辑：如"高收入"的定义需要了解业务

**提升准确率方案**

1. **Few-shot 示例**
   ```
   用户查询："最近一个月销售额超过 10 万的产品"
   SQL：SELECT * FROM products WHERE sale_amount > 100000
        AND created_date >= DATE_SUB(NOW(), INTERVAL 1 MONTH)
   ```
   - 提供 3-5 个相似的例子
   - 模型更容易理解模式

2. **Schema 信息**
   - 提供表结构描述
   - 字段名、类型、含义
   ```
   表：products
     - product_id: 产品 ID
     - name: 产品名称
     - sale_amount: 销售额（单位：元）
     - created_date: 创建日期
   ```

3. **SQL 验证**
   - 生成 SQL 后不直接执行
   - 先验证语法（sqlparse）
   - 再验证逻辑（预估结果数）

4. **Retrieval-Augmented SQL**
   - 从历史查询中检索相似的 SQL
   - 作为示例或参考
   - 提高准确率

5. **分步生成**
   - 第一步：理解意图，生成中间表示
   - 第二步：从中间表示生成 SQL
   - 第三步：验证和优化

### Query 润色

**目的**
- 将用户的模糊查询转为清晰的中间表示
- 补全缺失的上下文
- 消除歧义

**实现方法**
```
原始查询：
"最近的销售记录"

润色后：
"显示过去 7 天内的销售记录，包括产品名、销售额、日期，
按日期降序排列"
```

**何时需要**
- 用户查询不清楚
- 需要与历史上下文关联
- 需要执行多步查询

### Code Generation

**确保准确性**

1. **Unit Test 验证**
   - 生成代码后，生成简单的测试用例
   - 运行测试看是否通过
   - 失败则重新生成

2. **静态分析**
   - 语法检查（linter）
   - 类型检查
   - 安全检查（避免注入）

3. **沙箱执行**
   - 在隔离环境中运行代码
   - 防止恶意代码
   - 验证输出

4. **微调专用模型**
   - CodeLlama 等代码专用模型
   - 或对通用模型进行代码微调
   - 显著提升准确率

5. **渐进式生成**
   - 先生成函数签名
   - 再填充实现
   - 再生成测试

### Replan 机制

**什么是？**
- 执行中发现计划不可行
- 动态重新规划

**何时触发？**
- 工具调用失败
- 中间结果不符合预期
- 发现新的约束或信息

**怎么实现？**
```python
# 伪代码
while not task_complete:
    plan = agent.plan(task, context)
    result = agent.execute(plan)

    if result.success:
        task_complete = True
    else:
        # 触发 Replan
        context.add_failure_info(result.error)
        agent.reflect()  # 反思失败原因
        # 循环继续，会生成新的计划
```

**高频题**：你的系统怎么量化工作流的效果？
- 预期答案：
  - 每个工作流的成功率
  - 人工评估的质量得分
  - A/B 测试对比效果
  - 成本（Token 数、API 调用数）

## 第六层：Agent 系统设计（核心！）

### 记忆系统

**短期记忆**
- 组成：当前对话的 messages 列表
- 生命周期：单次对话
- 存储：内存或数据库
- 作用：提供上下文

**长期记忆**
- 存储形式：向量库 + 文本总结
- 粒度：
  - 会话级：完整对话摘要
  - 用户级：用户的偏好、历史
  - 全局级：常见问题、已解决的问题

- 检索策略：
  - 相似性搜索：向量相似度
  - 时间衰减：最近的记忆权重高
  - 频率衰减：常被检索的记忆权重高

**高频题**：记忆怎么存？粒度多大？会不会爆炸？
- 存储：向量库（如 Qdrant）存储向量 + 文本，SQL 存储元数据
- 粒度：通常会话级或 N 轮对话级
- 防爆炸：
  - 定期清理（过期记忆删除）
  - 优先级清理（低重要性先删）
  - 分层存储（热数据 in-memory，冷数据 disk）
  - 总大小限制

**实现示例**
```
新对话开始
  → 检索相关的长期记忆
  → 插入到系统提示或上下文中
  → 对话过程中不断更新短期记忆
  → 对话结束时
    → 总结关键信息
    → 存入长期记忆向量库
    → 设置过期时间
```

### Function Call 实现

**完整流程**

1. **工具定义阶段**
   ```json
   {
     "name": "search_products",
     "description": "搜索产品信息",
     "parameters": {
       "type": "object",
       "properties": {
         "keyword": {"type": "string", "description": "搜索关键词"},
         "category": {"type": "string", "description": "产品类别"}
       },
       "required": ["keyword"]
     }
   }
   ```

2. **调用阶段**
   - 模型分析用户查询
   - 决定调用哪个工具
   - 输出结构化的工具调用（JSON）

3. **执行阶段**
   - 系统解析 JSON
   - 验证参数
   - 执行实际函数
   - 返回结果

4. **反馈阶段**
   - 将工具结果返回给模型
   - 模型继续推理
   - 可能调用更多工具或生成最终答案

**错误处理**
```
工具调用失败
  ↓ 记录错误信息
  ↓ 返回给模型："调用失败，原因：..."
  ↓ 模型选择：
    - 重试（参数调整）
    - 换个工具
    - 告诉用户无法完成
```

### Single-Agent vs Multi-Agent

#### **Single-Agent**
- 一个模型处理所有任务
- 架构简单
- 延迟低
- 但能力受限

**适用场景**
- 简单问题
- 对延迟敏感
- 资源受限

#### **Multi-Agent**
- 不同 Agent 不同专长（Specialist）
- 通过对话、投票、协作解决问题

**架构设计**

1. **顺序执行**
   ```
   Agent A（理解问题）
     ↓ 结果
   Agent B（检索相关信息）
     ↓ 结果
   Agent C（生成答案）
   ```
   - 简单、易控制
   - 但串行，延迟高

2. **并行执行**
   ```
   Agent A ┐
   Agent B ├ 并行 → 融合结果
   Agent C ┘
   ```
   - 延迟低
   - 需要融合策略

3. **树形执行**
   ```
   Master Agent（主控）
     ├─ Worker A
     ├─ Worker B
     └─ Worker C
   Master 分配任务、聚合结果
   ```
   - 灵活
   - 复杂度高

**如何设计 Agent 间的通信？**
- 显式消息：A 发消息给 B（显式路由）
- 隐式消息：通过共享状态（消息队列、数据库）
- 协议：定义消息格式、错误处理

**投票机制**
- 多个 Agent 独立生成答案
- 投票选择最多支持的答案
- 提高准确率但成本 3 倍

### 反思机制

**什么是？**
- Agent 自我批评和改进
- 生成答案后，用新的提示让模型审视答案

**为什么需要？**
- 提高准确率（自我纠正）
- 提高可靠性（发现潜在问题）

**实现**
```python
# 第一步：生成答案
answer = agent.generate(question)

# 第二步：反思
reflection_prompt = f"""
原始问题：{question}
生成的答案：{answer}

请批评这个答案：
1. 是否准确回答了问题？
2. 是否有逻辑错误？
3. 是否有信息遗漏？
4. 如何改进？
"""
criticism = model.generate(reflection_prompt)

# 第三步：改进
if criticism.needs_improvement:
    answer = agent.generate(question, feedback=criticism)
```

**何时使用**
- 复杂问题（多步推理）
- 高精度要求
- Token 成本可接受

### 端到端延迟优化

**延迟公式**
```
总延迟 = 规划时间 + 执行时间 + 生成时间 + 网络延迟
       ≈ N 个 Agent turn × (平均 LLM 延迟 + 工具延迟)
```

**优化方向**

1. **减少 Agent 轮数**
   - 合理的初始规划
   - 减少不必要的反思
   - 目标：1-3 轮

2. **并行调用工具**
   - 多个工具同时执行
   - 减少串行等待
   - 示例：同时检索 3 个数据源

3. **模型选择**
   - 简单任务用小模型（快、便宜）
   - 复杂任务用大模型（准确）
   - 示例：GPT-3.5 用于规划，GPT-4 用于生成

4. **Prompt Caching**
   - 缓存系统提示
   - 缓存长文档
   - 减少 Token 传输

5. **流式响应**
   - 不等待全部完成
   - 逐步返回结果
   - 改善用户体验

6. **预计算和缓存**
   - 常见查询预计算答案
   - 相似查询复用结果

**监控指标**
```
单步延迟：
  - LLM 调用延迟（包括网络）
  - 工具执行延迟
  - 响应生成延迟

整体延迟：
  - P50、P95、P99 百分位
  - 峰值延迟
```

### 高可用和稳健性

**容错机制**

1. **工具失败重试**
   ```python
   for attempt in range(3):  # 最多重试 3 次
       try:
           result = tool.call(params)
           return result
       except ToolError as e:
           if attempt < 2:
               # 参数调整后重试
               params = adjust_params(params, e)
               continue
           else:
               # 放弃，返回错误或降级
               return fallback_result()
   ```

2. **模型降级**
   - 主模型（GPT-4）失败
   - 自动切换到备用模型（GPT-3.5）
   - 牺牲精度保证可用性

3. **超时处理**
   ```python
   with timeout(seconds=30):
       result = agent.execute(task)
   ```

4. **人工介入**
   - Agent 无法完成时转人工
   - 人工处理 + 反馈给系统学习

**监控和告警**

```
实时监控：
  - 成功率（百分比）
  - 平均延迟（毫秒）
  - 错误分类计数

告警规则：
  - 成功率 < 95% → 告警
  - P99 延迟 > 10s → 告警
  - 错误日志增长 > 100/分钟 → 告警
```

**并发处理**

1. **限流（Rate Limiting）**
   - 防止过载
   - 策略：Token bucket、Sliding window

2. **队列管理**
   - 请求入队
   - 工作线程池处理
   - 防止内存爆炸

3. **动态扩容**
   - 监控 CPU、显存使用
   - 自动增加/减少工作线程
   - Kubernetes 场景：自动扩容 Pod

### 高频题：你最大的难题是什么？

**常见答案+解决方案**

1. **幻觉问题**
   - 原因：参数知识 + 检索不当
   - 解决：RAG + 引用标注 + 验证机制
   - 效果：准确率从 75% → 92%

2. **延迟高**
   - 原因：多轮规划、工具串行
   - 解决：并行化工具调用、减少轮数
   - 效果：延迟从 5s → 1.5s

3. **成功率低**
   - 原因：工具数量少、提示词不好
   - 解决：增加工具、优化 prompt、多 Agent 投票
   - 效果：成功率从 80% → 95%

4. **成本高**
   - 原因：频繁调用大模型
   - 解决：多模型混用、缓存、量化
   - 效果：成本降低 60%

## 第七层：系统设计题（综合考察）

### 场景 1：游戏社区客服助手

**核心需求**
- 回答游戏相关问题（规则、玩法、充值）
- 理解游戏黑话（"抽卡""Buff"等）
- 利用公司内部文档（FAQ、运营指南）

**架构设计**

```
用户提问
  ↓
意图分类 Agent
  ├─ 简单问题？→ 直接查 FAQ 库
  └─ 复杂问题？→ 多路检索

多路检索
  ├─ BM25：关键词匹配（黑话库）
  ├─ 向量检索：语义相似（通用文档）
  └─ 知识图谱：关系推断（游戏规则）

上下文组织
  + 游戏黑话词表
  + 相关规则文档
  + 历史问题参考

生成答案
  + 引用出处
  + 不确定时转人工

用户满意度反馈
  → 低分答案进入改进队列
```

**关键技术点**
- 黑话识别：微调 NER 模型识别黑话
- 知识图谱：游戏物品、角色、关系
- 多语言支持：游戏可能有多语言社区
- 实时性：新版本、新活动及时更新文档

**成本优化**
- 热点问题缓存（FAQ）
- 用小模型做意图分类
- 只在必要时调用大模型

### 场景 2：长篇小说文档切割

**挑战**
- 保留情节连贯性
- 避免切割关键转折
- 处理复杂结构（多线叙事）

**方案**

```
第一步：结构分析
  - 识别章节、段落、句子层级
  - 提取关键情节转折点

第二步：语义分割
  - 按段落切割（而不是固定大小）
  - 保留上下文关系

第三步：质量检查
  - 验证每个片段的独立性
  - 检查关键信息是否完整

第四步：向量化
  - 生成片段摘要
  - 计算向量用于检索

存储
  - 原文段落 + 摘要 + 向量 + 元数据（章节、页码）
```

**技术细节**
- 关键情节识别：用关键词 + NLP 检测
- 上下文保留：包含前后段落摘要
- 分层检索：先按章节，再精细搜索

### 场景 3：论文翻译并保留格式

**难点**
- 保留数学公式、图表位置
- 维护引用编号一致性
- 处理特殊排版（代码块、表格）

**方案**

```
第一步：结构提取
  - 识别标题、公式、图表、表格、代码
  - 标记其位置和编号
  - 提取纯文本内容

第二步：翻译
  - 分段翻译（段落为单位）
  - 使用领域术语词典（如 ML 论文术语）
  - 保留引用编号 [1]、[2]

第三步：重组
  - 按原始结构重建文档
  - 公式、图表位置复原
  - 引用编号校验

第四步：质量检查
  - 人工审阅关键部分
  - 公式语法检查
  - 格式完整性验证
```

**技术栈**
- 文档解析：PyPDF2 / pdfplumber 提取结构
- 翻译引擎：调用翻译 API（支持设置术语表）
- 文档重建：ReportLab / python-docx 生成
- OCR：处理扫描版论文

### 场景 4：结构化+非结构化数据分析

**典型问题**
- 用户在数据库中搜索符合条件的记录
- 同时需要分析非结构化数据（文本、评论）

**架构**

```
用户查询："找出过去 3 月销售 > 50 万的商家，
           并分析其用户评论情况"

路径 1：结构化数据（SQL）
  SQL Agent
    ↓
  SELECT 商家 FROM sales
  WHERE date >= DATE_SUB(NOW(), INTERVAL 3 MONTH)
  AND sales > 500000
  ↓ 返回 50 个商家 ID

路径 2：非结构化数据（NLP）
  对每个商家的评论进行：
  - 情感分析
  - 关键词提取
  - 主题聚类

融合
  - 结构化数据排序
  - + 非结构化数据的情感得分
  - = 最终排序结果
```

**技术要点**
- SQL Agent：利用 Text-to-SQL
- 批量 NLP 处理：并行化以降低延迟
- 向量化：评论转向量用于聚类
- 结果融合：加权组合多个得分

**优化**
- 缓存热门商家的分析结果
- 异步处理 NLP（不阻塞 SQL 结果）
- 流式返回部分结果

## 第八层：当下趋势（为什么这样做）

### LLM 应用的趋势

**从 Chat 到 Agent**
- 第一阶段：Chat 应用（一问一答）
- 第二阶段：Agent（多步规划执行）
- 第三阶段：Embodied AI（与物理世界交互）

**Agentic AI 的核心**
- 模型主动规划
- 选择合适工具
- 持续执行和反馈
- 自我改进能力

**垂直化微调**
- 通用模型基础能力强
- 但特定领域表现不如微调模型
- 趋势：通用 + 垂直微调结合

**成本优化**
- 多模型混用（大小模型结合）
- 量化技术（减少显存）
- 知识蒸馏（教小模型）
- 缓存和复用

**可靠性提升**
- 验证机制（生成后验证）
- 可解释性（输出推理过程）
- 人机协作（不确定时转人工）

### WebRTC vs WebSocket

**WebSocket**
- 双向通信通道
- 基于 TCP，单个连接
- 延迟：中等（取决于网络）
- 适合：简单的数据交互

**WebRTC**
- 点对点（P2P）通信
- 低延迟（UDP 优化）
- 支持媒体流（音频、视频）
- 内置 NAT 穿透（STUN/TURN）

**Agent 应用中的选择**

- **简单对话**：WebSocket
- **实时流式传输**：WebRTC
- **包含音视频**：WebRTC
- **多方通信**：WebRTC（有 MCU/SFU）

**示例**
```
用户 → WebSocket → Agent 服务
          ↓
      处理和规划
          ↓
     WebSocket → 流式返回结果

如果需要音频交互：
用户（麦克风）→ WebRTC → 音频处理
                          ↓
                      文字转换 → Agent → 合成回复
                                        → 返回音频
```

## 附录：常用命令和代码片段

### LangChain 快速开始

```python
from langchain.chat_models import ChatOpenAI
from langchain.agents import initialize_agent, Tool
from langchain.tools import tool

# 定义工具
@tool
def search_products(query: str) -> str:
    """搜索产品"""
    return f"找到关于 {query} 的产品"

# 初始化 Agent
tools = [search_products]
llm = ChatOpenAI(model="gpt-4")
agent = initialize_agent(tools, llm, agent="zero-shot-react-description")

# 执行
result = agent.run("帮我找最好的笔记本")
```

### RAG 检索示例

```python
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Qdrant
from langchain.document_loaders import PDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 加载文档
loader = PDFLoader("document.pdf")
docs = loader.load()

# 分割
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=50
)
splits = splitter.split_documents(docs)

# 向量化和存储
embeddings = OpenAIEmbeddings()
vectorstore = Qdrant.from_documents(
    splits,
    embeddings,
    path="./qdrant_storage"
)

# 检索
query = "如何使用这个产品？"
results = vectorstore.similarity_search(query, k=3)
```

### Function Call 示例

```python
from openai import OpenAI

client = OpenAI()

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取天气信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {"type": "string"},
                    "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                },
                "required": ["location"]
            }
        }
    }
]

response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "北京天气怎么样？"}],
    tools=tools,
    tool_choice="auto"
)
```
