## ✅ 今日学习进度

### 第 1 部分：核心概念（30-45分钟）

#### 学习内容 1️⃣：Text-to-SQL 的 3 个关键难点
- [x] **难点 1：Schema 理解障碍**
  - 原因：表字段众多、命名不统一、关系复杂
  - 影响：准确率下降 30-40%
  - 关键：清晰的 Schema 文档是解决方案

- [x] **难点 2：语义-SQL 转换困难**
  - 原因：自然语言歧义、业务逻辑复杂
  - 影响：准确率下降 20-30%
  - 关键：Query 改写和 Few-shot 示例

- [x] **难点 3：复杂逻辑编程**
  - 原因：需要 CTE、窗口函数、嵌套查询
  - 影响：准确率下降 40-60%（复杂查询）
  - 关键：递进式的 Few-shot 示例

#### 学习内容 2️⃣：Schema 设计的 5 个最佳实践
- [x] **实践 1：清晰的 Schema 文档**
  - 必须包含：表说明、字段说明、关系说明、常用查询
  - 关键：业务规则和陷阱要明确写出

- [x] **实践 2：统一的字段命名规范**
  - ID 字段：{entity}_id
  - 日期字段：统一后缀（_date, _at）
  - 布尔值：is_ 前缀
  - 金额：amount 或 price
  - 成效：准确率提升 10-15%

- [x] **实践 3：明确的关系说明**
  - 1:N 关系、N:N 关系、自关联
  - 包含常用 JOIN 示例

- [x] **实践 4：业务规则和陷阱**
  - cancelled 订单的处理
  - 时间字段的含义（order_date vs shipped_date vs delivered_date）
  - 价格字段的选择（order_items.unit_price vs products.price）

- [x] **实践 5：示例查询集合**
  - 预制常见查询的 SQL 模板
  - 为 Few-shot 学习铺路

#### 学习内容 3️⃣：Few-shot 示例的构造方法
- [x] **4 个关键要素**
  1. **多样性**：覆盖 simple → medium → hard
  2. **清晰的注释**：解释关键逻辑
  3. **涵盖业务规则**：体现 cancelled 过滤等规则
  4. **逐步递进**：建立信心和学习效果

- [x] **Few-shot 的心理学**
  - 前面简单例子让 LLM "预热"
  - 中间例子建立信心
  - 复杂例子说明能力
  - 结果：准确率提升 15-25%

### 第 2 部分：技术细节（45-60分钟）

#### 学习内容 1️⃣：SQL 验证和纠错策略
- [x] **策略 1：语法验证**
  - 检查必要关键字（SELECT）
  - 检查 JOIN 和 ON 一致性
  - 检查括号、引号平衡
  - 成本：最低，用正则表达式

- [x] **策略 2：Schema 验证**
  - 检查表是否存在
  - 检查字段是否存在
  - 检查 JOIN 引用的表
  - 成本：低，用字典查询

- [x] **策略 3：逻辑验证（试运行）**
  - 实际在数据库执行 SQL
  - 检查是否真的能跑
  - 成本：最高（实际查询成本），但最准确

- [x] **策略 4：业务规则验证**
  - 检查是否过滤了 cancelled 订单
  - 检查是否用了正确的价格字段
  - 检查是否用了正确的时间字段
  - 成本：低，用规则引擎

#### 学习内容 2️⃣：Query 改写（处理歧义）
- [x] **问题识别**
  - 自然语言有歧义（"经常"可能是次数、频率、消费额等）
  - 一个问题可能有多个有效 SQL

- [x] **解决方案**
  - 让 LLM 识别歧义
  - 生成多个明确的问题版本
  - 选择最可能的版本
  - 然后转换成 SQL

#### 学习内容 3️⃣：成本优化
- [x] **优化 1：Prompt 缓存**
  - Schema 文档被缓存，不重复计算
  - 成本降低：97%（100 次查询）

- [x] **优化 2：SQL 模板库**
  - 常见查询用预制模板
  - 避免重复调用 LLM
  - 成本节省：70-80%

- [x] **优化 3：分层策略**
  - 简单问题：用模板（成本 ↓ 95%）
  - 中等问题：用缓存 LLM（成本 ↓ 80%）
  - 复杂问题：用缓存 + 验证（成本 ↓ 30%）
