## 模块 1：Transformer 架构概览

### 1.1 背景和问题

**问题**：什么模型能处理长文本，并且能并行处理？

**答案**：Transformer（2017 年"Attention is All You Need"论文提出）

### 1.2 为什么需要 Transformer？

```
RNN 的问题：
1. 必须顺序处理（不能并行）→ 训练慢
2. 长距离依赖问题 → 远距离信息衰减

Transformer 的优势：
1. 完全并行化 → 训练快 10 倍以上
2. 直接连接任意两个位置 → 无衰减
3. 易于堆叠深层网络 → 模型更强
```

### 1.3 Transformer 的整体架构

```
输入文本
  ↓
Embedding（词嵌入）
  ↓
Positional Encoding（位置编码）
  ↓
Transformer Encoder（多层）
  ├─ Attention 层（学习词间关系）
  └─ FFN 层（学习复杂特征）
  ↓
Transformer Decoder（多层）
  ├─ Attention 层
  ├─ Cross-Attention 层
  └─ FFN 层
  ↓
输出层（线性投影 + Softmax）
  ↓
输出（概率分布或生成的词）
```

### 1.4 Encoder vs Decoder

| 特性 | Encoder | Decoder |
|------|---------|---------|
| **视野** | 可以看全部上下文 | 只能看之前的词（用掩码） |
| **任务** | 理解任务（分类、提取） | 生成任务（翻译、对话） |
| **代表** | BERT | GPT、T5-Decoder |
| **层数** | N 层 | N 层 |

**关键点**：ChatGPT 是 Decoder-only 模型（只能看之前的词，按顺序生成）

## 模块 2：Self-Attention 机制（核心）

### 2.1 问题引入

**核心问题**：一个词怎么关注其他词？

**答案**：通过计算相似度，然后加权求和

### 2.2 Self-Attention 的三步骤

#### **第 1 步：计算 Q、K、V**

```
原始输入：x（词的 embedding 向量）
三个线性变换矩阵：W_Q, W_K, W_V

计算：
Q = x · W_Q  (Query - "我在找什么")
K = x · W_K  (Key - "我是谁")
V = x · W_V  (Value - "我的信息")

直观理解：
- Q：这个词的"查询"向量
- K：这个词的"索引"向量（用来被匹配）
- V：这个词的"信息"向量（真实内容）
```

**例子**：
```
句子："The cat sat on the mat"

单词"sat"的 embedding 输入
  ↓
乘以 W_Q → Q_sat = "我在找'发生了什么'的相关词"
乘以 W_K → K_sat = "我代表'坐'这个动作"
乘以 W_V → V_sat = "我的完整含义信息"
```

#### **第 2 步：计算注意力权重**

```
公式：
Attention_weights = softmax(Q · K^T / √d_k)

步骤解析：

1. Q · K^T：计算相似度
   - Q 的维度：(seq_len, d_k)
   - K^T 的维度：(d_k, seq_len)
   - 结果：(seq_len, seq_len) 的相似度矩阵

   例如"sat"的 Q 与所有词的 K 点积：
   [sat-the, sat-cat, sat-sat, sat-on, sat-the, sat-mat]
   = [0.3, 0.5, 2.1, 0.8, 0.3, 0.6]

2. 除以 √d_k：缩放
   为什么？防止数值过大导致梯度消失
   d_k = 64 时，√d_k = 8

   缩放后：[0.038, 0.063, 0.263, 0.1, 0.038, 0.075]

3. softmax：转换为概率
   结果：[0.10, 0.12, 0.45, 0.15, 0.10, 0.08]

   意义："sat"对各词的关注度
```

#### **第 3 步：加权求和得到输出**

```
公式：
Output = Attention_weights · V

步骤：

假设 V 矩阵（所有词的 V 向量堆叠）：
V = [V_the, V_cat, V_sat, V_on, V_the, V_mat]
    = [v1, v2, v3, v4, v5, v6]

权重（从第 2 步）：
w = [0.10, 0.12, 0.45, 0.15, 0.10, 0.08]

输出向量：
Output_sat = 0.10·v1 + 0.12·v2 + 0.45·v3 + 0.15·v4 + 0.10·v5 + 0.08·v6

含义：
"sat"的新表示 = 45% 来自自己 + 15% 来自"on" + 12% 来自"cat" + ...
```

### 2.3 为什么除以 √d_k？

```
问题：Q · K^T 的数值会很大，导致什么问题？

答案：
1. 数值过大 → softmax 会近似 one-hot → 梯度消失
2. 缩放后数值适中 → softmax 分布平缓 → 梯度更新正常

例子：
未缩放：Q·K^T = [30, 25, 50, 20, 30, 25]
        softmax = [0, 0, 1, 0, 0, 0]（几乎全是最大值）
        梯度：只有最大值位置有梯度，其他都是 0 → 无法学

缩放后：Q·K^T / √64 = [0.3, 0.25, 0.5, 0.2, 0.3, 0.25]
        softmax = [0.15, 0.14, 0.24, 0.12, 0.15, 0.14]（分布平缓）
        梯度：每个位置都有梯度 → 正常学
```

### 2.4 完整公式

```
Attention(Q, K, V) = softmax(Q · K^T / √d_k) · V

这就是 Self-Attention 的完整公式
```

## 模块 3：Multi-Head Attention（多头注意力）

### 3.1 问题：为什么需要多头？

```
单头的限制：
- 一个 Attention 头只学一种关系
- 比如只关注"名词和动词"的关系
- 但模型需要同时学多种关系：语法、语义、指代等

解决方案：
- 用多个 Attention 头（通常 8-16 个）
- 每个头学不同的关系
- 最后合并所有头的结果
```

### 3.2 多头的工作流程

```
输入：x（seq_len × d_model）

第 1 步：分割
  原维度 d_model = 512
  分成 8 个头
  每个头的维度 d_k = 512 / 8 = 64

  x 分割成 8 份：
  x_1(seq × 64), x_2(seq × 64), ..., x_8(seq × 64)

第 2 步：并行计算 8 个 Attention
  head_1 = Attention(x_1 W_Q, x_1 W_K, x_1 W_V)
  head_2 = Attention(x_2 W_Q, x_2 W_K, x_2 W_V)
  ...
  head_8 = Attention(x_8 W_Q, x_8 W_K, x_8 W_V)

第 3 步：拼接
  concat_output = [head_1 || head_2 || ... || head_8]
  维度：seq_len × 512

第 4 步：线性投影（W_O）
  MultiHead_output = concat_output · W_O
  W_O：512 × 512 矩阵
  作用：融合 8 个头的信息
```

### 3.3 为什么是 8 个头？

```
权衡分析：

2 个头：
  优点：计算快
  缺点：无法学多种关系

8 个头：
  优点：够多(学多种关系) + 不太多(计算可接受)
  ✓ 经验值最优

16 个头：
  优点：更多关系类型
  缺点：计算量增加，效果提升有限

64 个头：
  缺点：过度细分，有些头不发挥作用
```

## 模块 4：位置编码（Positional Encoding）

### 4.1 问题：Attention 无法感知位置

```
为什么？
- Attention 是对称操作：Q · K^T
- 位置 1 和位置 2 的 K 计算出的相似度
- 和位置 2 和位置 1 计算的相似度相同
- 所以位置信息在 Attention 中被抹除了

例子：
短语"狗咬人"和"人咬狗"
- 词汇相同，但 Attention 无法区分顺序
- 需要额外的位置信息来区分
```

### 4.2 解决方案：位置编码

```
方法：
在 embedding 中加入位置信息

流程：
Word embedding e_word（维度 512）
  +
Position encoding e_pos（维度 512）
  =
Final embedding（维度 512，融合了词和位置信息）
```

### 4.3 位置编码的公式

```
PE(pos, 2i) = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))

其中：
- pos：词在句子中的位置（0, 1, 2, ...）
- i：维度的索引（0, 1, 2, ..., d_model/2-1）
- d_model：embedding 维度（通常 512）

例子（d_model=8）：
位置 0：[sin(0), cos(0), sin(0), cos(0), ...]
       = [0, 1, 0, 1, 0, 1, 0, 1]

位置 1：[sin(1/1), cos(1/1), sin(1/100), cos(1/100), ...]
       = [0.84, 0.54, 0.01, 1.0, ...]

位置 2：[sin(2/1), cos(2/1), sin(2/100), cos(2/100), ...]
       = [0.91, -0.42, 0.02, 1.0, ...]
```

### 4.4 为什么用 sin/cos？

```
优势 1：周期性
- 低维度（i 小）：高频率变化 → 区分近距离位置
- 高维度（i 大）：低频率变化 → 区分远距离位置
- 这样模型能学到相对位置的概念

优势 2：支持任意长度
- 公式对任意位置都有定义
- 不需要为更长的序列重新训练
- RNN 就没这个优势

优势 3：相对位置编码
- PE(pos + k) 可以表示为 PE(pos) 的线性组合
- 让模型学到"相对位置"而不是"绝对位置"
```

## 模块 2：Feed-Forward Network

### 2.1 问题引入

**问题**：Transformer 中只有 Attention 够吗？

**答案**：不够。除了"交互"，还需要"深化"

### 2.2 FFN 的结构

```
FFN(x) = Max(0, x · W_1 + b_1) · W_2 + b_2

也称 Position-wise Feed-Forward Network

结构：
输入 x（512 维）
  ↓
线性变换 1：升维（512 → 2048）
  x_hidden = x · W_1 + b_1

  ↓
ReLU 激活：引入非线性
  x_activated = max(0, x_hidden)

  ↓
线性变换 2：降维（2048 → 512）
  output = x_activated · W_2 + b_2

  ↓
输出（512 维）
```

### 2.3 关键理解

#### **为什么要升维？**

```
原因：在高维空间有更多自由度学复杂特征

类比：
- 在 2 维空间，只能学简单的线条关系
- 在 100 维空间，能学复杂的非线性关系
- FFN 升维到 2048 维，让模型有空间学复杂变换
```

#### **为什么要用 ReLU？**

```
作用：引入非线性

如果只用线性变换：
  output = (x · W_1) · W_2 = x · (W_1 · W_2)
  结果：仍是线性变换！参数多也没用

使用 ReLU：
  x_hidden = ReLU(x · W_1)
  output = x_hidden · W_2
  结果：非线性变换，能学复杂函数

ReLU 的作用：
  ReLU(x) = max(0, x)
  - 激活某些维度（x > 0）
  - 关闭其他维度（x ≤ 0）
  - 创造了"选择性激活"，产生非线性
```

#### **为什么要降维？**

```
升维学特征 → 降维融合特征

结果需要回到原维度（512），因为：
- Transformer 中每层都需要相同维度
- 便于叠加多层（残差连接）
- 保持计算图的一致性
```

### 2.4 Attention vs FFN

```
Attention 层：
- 作用：全局交互（开会议）
- 每个词与其他所有词互动
- 学习"词之间的关系"
- 矩阵计算：Q·K·V

FFN 层：
- 作用：局部深化（单独思考）
- 每个词单独处理（独立变换）
- 学习"复杂特征"
- 矩阵计算：线性→激活→线性

完整块：Attention + FFN
- Attention 是"沟通阶段"
- FFN 是"思考阶段"
- 两个都需要
```

### 2.5 深度理解

#### **FFN 只处理单个词，怎么还有用？**

```
关键点：
- FFN 输入的每个词已经包含其他词的信息（来自 Attention）
- FFN 对这些"融合后的词向量"进行深层变换
- 学到的是"如何基于融合信息做进一步处理"

时间序列：
第 1 步（Attention）：词 1、2、3 互动，融合信息
  输入：[e_1, e_2, e_3]（独立向量）
  输出：[e_1', e_2', e_3']（融合向量）

第 2 步（FFN）：每个融合后的词独立深化
  输入：[e_1', e_2', e_3']
  变换：升维→非线性→降维
  输出：[f_1', f_2', f_3']（深化向量）
```

#### **没有 FFN 会怎样？**

```
后果 1：只有交互，没有深化
- 模型只能"信息汇聚"
- 无法学到复杂的特征组合

后果 2：表达能力受限
- Attention 是线性组合
- 多层 Attention 堆叠仍主要是线性操作
- 需要非线性才能达到通用逼近

结论：FFN 提供了"学复杂函数"的能力
```

## 模块 3：大模型训练流程（三阶段）

### 3.1 总体框架

```
现代大语言模型的诞生需要三个阶段：

第 1 阶段：预训练（Pretraining）
   ↓ 产出：能"理解"文本的模型

第 2 阶段：指令微调（Instruction Tuning）
   ↓ 产出：能"听指令"的模型

第 3 阶段：强化学习对齐（RLHF）
   ↓ 产出：能"生成高质量"的模型

最终：ChatGPT、Claude 这样的聊天助手
```

### 3.2 阶段 1：预训练（Pretraining）

#### **任务**

```
Next Token Prediction（预测下一个词）

例子：
输入：The cat sat on the ___
预测：mat

输入：I love ___
预测：you

模型的工作：看前面的词，预测下一个词
```

#### **数据**

```
数据量：TB 级（几百万 GB）
质量：高质量网页、书籍、代码

例子：
- 互联网爬虫数据（Common Crawl）
- 维基百科
- GitHub 代码
- Books 数据集

特点：无标注，只需要原始文本
```

#### **模型学到什么**

在预测下一个词的过程中，模型被迫学到：

```
1. 语言基础
   - 语法规则
   - 词汇含义
   - 短语搭配

2. 世界常识
   - 人物、地点、事件
   - 因果关系
   - 常识推理

3. 问题求解
   - 简单推理
   - 模式识别
   - 逻辑推导

4. 编程知识
   - 代码语法
   - 算法逻辑
   - API 使用

为什么能学到这么多？
因为要"预测下一个词"，必须理解前文的含义
```

#### **时间和成本**

```
时间：数周到数个月
成本：百万级 GPU 小时
例子：GPT-3 预训练花费 3-6 个月

局限：
- 模型不知道怎么"听指令"
- 只会"继续文本"，不会"回答问题"
```

### 3.3 阶段 2：指令微调（Instruction Tuning）

#### **问题**

```
预训练模型知道很多东西，但不会听指令

例子：
指令：翻译成中文："Hello world"
预训练模型可能的输出：
  Hello world is a famous...
  （继续英文，而不是翻译）

或者：
指令：解决数学题：2 + 2 = ?
预训练模型可能的输出：
  2 + 2 = 4 is the answer to...
  （虽然有正确答案，但格式不对）
```

#### **解决方案**

```
用指令-响应对微调模型

数据：
- 手工标注（高质量）
- 指令：用户的问题
- 响应：期望的回答
- 数量：几千到几万对

例子：
<指令> 翻译成中文："Hello world"
<响应> 你好 世界

<指令> 解决：2 + 2 = ?
<响应> 4

<指令> 写一首诗关于春天
<响应> 春风拂过江南岸，
        绿芽萌发新生命，
        ...
```

#### **训练过程**

```
1. 给模型一条指令
2. 模型生成响应
3. 对比期望的响应
4. 计算损失
5. 反向传播更新参数
6. 重复...

结果：
- 模型学会了"看到指令就执行对应操作"
- 但质量仍不稳定
  - 有时很好
  - 有时会胡编
  - 有时被骗
```

#### **时间和成本**

```
时间：数小时到数天
成本：低（相对预训练）
为什么快？因为数据少（几万 vs TB 级）

对比：
预训练：3 个月，100 万 GPU 小时
指令微调：1 天，1000 GPU 小时（快 1000 倍！）
```

### 3.4 阶段 3：强化学习对齐（RLHF）

#### **问题**

```
指令微调模型仍有问题：

1. 有时候胡编（称为"幻觉"）
   输入：介绍一下星球大战的演员
   输出：Luke Skywalker 饰演者是...[虚构名字]

2. 不够安全
   输入：怎样制造炸弹？
   输出：第 1 步...（直接给出危险信息）

3. 不够认真
   输入：解释量子力学
   输出：就是...比较难。（敷衍）

怎样解决？
```

#### **解决方案：RLHF（Reinforcement Learning from Human Feedback）**

**第 1 步：收集人工评分数据**

```
步骤：
1. 对同一个问题，让模型生成多个回答
   问题：介绍一下巴黎
   回答 A：巴黎是法国首都，有埃菲尔铁塔...
   回答 B：巴黎不知道
   回答 C：巴黎就是巴黎，很有名

2. 让人工评价哪个回答最好
   排序：A > C > B
   或评分：A(9分) > C(3分) > B(1分)

3. 收集数万条这样的评比
```

**第 2 步：训练奖励模型**

```
输入：(问题, 回答)
输出：这个回答有多好（0-1 之间的分数）

例子：
输入：("介绍巴黎", "巴黎是法国首都...")
输出：0.92（很好）

输入：("介绍巴黎", "我不知道")
输出：0.15（很差）

奖励模型学到了"什么是好回答"
```

**第 3 步：用强化学习优化原模型**

```
目标：最大化奖励

过程：
1. 原模型生成回答
2. 奖励模型评分
3. 计算奖励
4. 用强化学习算法（如 PPO）更新模型参数
   让模型倾向生成高奖励的回答

结果：
- 模型倾向生成"人类喜欢的"回答
- 更少胡编
- 更安全
- 更认真
```

#### **时间和成本**

```
时间：数小时到数天
成本：低（相对预训练）

对比：
预训练：3 个月
指令微调：1 天
RLHF：几小时
```

### 3.5 三阶段的本质

```
阶段 1 预训练：学基础知识
  数据：无标注，大规模
  任务：预测下一个词
  效果：通用但不听指令
  ↓

阶段 2 指令微调：学听指令
  数据：有标注，中等规模
  任务：指令-响应配对
  效果：听指令但质量不稳定
  ↓

阶段 3 RLHF：学生成高质量
  数据：人工评比，小规模
  任务：最大化人工奖励
  效果：高质量、安全、可靠
  ↓

最终模型：聪明 + 可靠 + 安全 + 听指令
```

## 模块 4：预训练和 Transformer 的关系

### 4.1 关键澄清

```
经常被混淆的概念：

Transformer 是什么？
  ✓ 神经网络的一个架构
  ✓ 一种模型结构
  ✓ 本身不含任何知识

预训练是什么？
  ✓ 一种训练过程
  ✓ 用数据训练模型
  ✓ 让模型学知识

两者的关系：
  Transformer 架构 + 预训练过程 = 有知识的大语言模型

类比：
  Transformer = 空的人脑
  预训练 = 读书学习
  结果 = 聪明的人
```

### 4.2 核心理解

```
预训练过程：
1. 初始化 Transformer（参数随机）
2. 把数据送入模型
3. 前向传播：计算预测
4. 计算损失：预测 vs 实际
5. 反向传播：计算梯度
6. 更新参数：梯度下降
7. 重复 2-6 数十亿次

最终：
  参数被优化到"好的值"
  这个模型就"学到"了预训练数据的知识
```

### 4.3 现代模型的架构

```
经常看到的说法：
"GPT-3 是 Transformer Decoder-only 模型"

含义：
- GPT-3 = Transformer 的 Decoder 部分
- 经过预训练（学知识）
- 经过指令微调（学听指令）
- 经过 RLHF（学生成高质量）

对比：
BERT：Transformer Encoder 部分
T5：完整 Transformer（Encoder + Decoder）
GPT：Transformer Decoder 部分
LLaMA：Transformer Decoder 部分（优化版）
```

|---|------|------|
| Transformer 本身含有知识吗？ | 不含，只是架构 | ⭐⭐⭐⭐⭐ |
| GPT-3 用的什么架构？ | Decoder-only Transformer | ⭐⭐⭐⭐⭐ |
| BERT 用什么架构？ | Encoder Transformer | ⭐⭐⭐⭐⭐ |

## 模块 5：Function Call 原理

### 5.1 问题引入

```
背景：
- 预训练模型只能预测下一个词
- 微调模型能回答问题但只生成文本
- 如何让模型"执行操作"（调用函数、API）？

答案：Function Call（函数调用）
```

### 5.2 Function Call 是什么？

```
定义：
Function Call = 模型决定调用哪个工具和参数

流程：
1. 定义工具（JSON Schema 描述）
2. 模型理解用户请求
3. 模型决定调用哪个工具
4. 系统执行工具
5. 模型基于结果组织答案

例子：
用户：明天天气怎样？
工具定义：{"name": "get_weather", "params": ["city", "date"]}

模型响应：调用 get_weather(city="北京", date="明天")
系统执行：get_weather("北京", "明天") = "晴朗，25°C"
模型生成：明天北京天气晴朗，25°C
```

### 5.3 Function Call 的训练

```
问题：预训练模型怎么知道怎样调用函数？

答案：在指令微调中训练！

步骤 1：收集函数调用数据
  指令："查询上海明天天气"
  响应：调用 get_weather(city="上海", date="明天")

步骤 2：微调时加入这些例子
  模型看到：很多"指令→函数调用"的例子
  学到：识别什么时候应该调用函数

步骤 3：结果
  模型获得了"调用函数的能力"
```

### 5.4 Function Call 和 Agent 的关系

```
Chatbot（没有 Function Call）：
  能回答问题，但只能生成文本
  无法执行实际操作

Agent（有 Function Call）：
  能理解任务
  能调用工具完成任务
  能基于结果继续推理

Function Call 是关键桥梁！
```

### 5.5 深度理解

```
为什么这么重要？

没有 FC：AI 只是文本生成机
  用户：帮我订飞机票
  AI：我可以帮你...（生成文本，但无法真正订票）

有 FC：AI 变成任务执行机
  用户：帮我订飞机票
  AI：调用 book_flight() → 执行订票 → 返回结果

FC 是从"Chatbot"到"Agent"的临界点
```
