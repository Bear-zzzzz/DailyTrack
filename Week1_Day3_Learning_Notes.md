## 📖 模块 1：LoRA 和 QLoRA 微调

### 1.1 核心问题

**背景**：预训练完成后，如何针对特定业务快速微调大模型？

**两个方案**：
1. Full Finetune：更新所有参数（效果最优，成本最高）
2. LoRA：只更新少量参数（成本低，效果 95%+）

### 1.2 LoRA 的核心洞察

**关键发现**：预训练模型的权重矩阵 W 具有低秩性质

```
含义：
- 权重矩阵虽然是 1000×1000 维
- 但大多数信息只集中在 8-32 维的低秩空间中
- 这意味着 99% 的信息可以用 1-2% 的参数表示
```

**为什么权重矩阵是低秩的？**
1. 数据的内在维度低（文本、图像的真实维度远小于特征维度）
2. 训练过程的隐式正则化（梯度下降倾向学习低秩解）
3. 模型过参数化导致冗余（参数远多于必要）

### 1.3 LoRA 的数学原理

**核心公式**：
```
原始：y = W·x
LoRA：y = (W + ΔW)·x = W·x + B·A·x

其中：
- W：预训练权重（冻结，不更新）
- ΔW = B·A：微调增量（学习参数）
- A：1000×r 矩阵（初始化为随机小数）
- B：r×1000 矩阵（初始化为零）
- r：秩（通常 8, 16, 32）
```

**参数数量对比**：
```
W：      1000×1000 = 100万参数
ΔW：     B·A = (r×1000) + (1000×r) ≈ 2000r 参数
         当 r=8 时：16000 参数（只是 W 的 1.6%！）
```

### 1.4 LoRA 的工作流程

```
时刻 T0（预训练完成）：
  ✓ 得到预训练权重 W
  ✓ 这是一个通用模型（如 GPT-3.5 70B 参数）

时刻 T1（微调准备）：
  ✓ 冻结 W（不动）
  ✓ 初始化 A 和 B（只有 1.4B 参数）

时刻 T2-T100（微调过程）：
  ✓ 前向传播：y = W·x + B·A·x
  ✓ 计算损失
  ✓ 反向传播：计算 dA 和 dB（但不计算 dW）
  ✓ 更新 A 和 B：A ← A - α·dA，B ← B - α·dB
  ✓ W 保持不变

时刻 T101（微调完成）：
  ✓ 最终模型 = W + B·A
  ✓ 参数数量：1.6%
  ✓ 效果：95-99% 的 Full Finetune
```

**关键理解**：LoRA 不改变 W，而是添加一个低秩增量 ΔW

### 1.5 参数来源解析

**W 从何而来？**
- 来源：预训练阶段通过梯度下降学出来
- 时间：花费数周 GPU 时间
- 方法：在 TB 级无标注文本上反复执行"前向→损失→反向→更新"
- 结果：每个参数都被更新数十亿次

**ΔW 从何而来？**
- 来源：微调阶段通过梯度下降学出来
- 时间：花费数小时 GPU 时间（1000 倍快）
- 方法：在小规模标注数据上反复执行"前向→损失→反向→更新"
- 结果：A 和 B 学到"新知识"，ΔW = B·A 是"新知识"的编码

### 1.6 三个关键对比

| 维度 | Full Finetune | LoRA | QLoRA |
|------|---|---|---|
| **参数数量** | 100%（70B）| 1.6%（1.4B）| 1.6%+量化 |
| **显存占用** | 很高 | 低 20-30 倍 | 低 40 倍 |
| **训练速度** | 慢（天级）| 快（小时级）| 最快 |
| **效果** | 100% | 95-99% | 95-99% |
| **成本** | ¥¥¥¥ | ¥ | ¥ |

### 1.7 LoRA 的秩（r）选择

```
r = 0：
  ΔW = 0，模型完全不变
  效果：和预训练模型一样

r = 8：
  参数 1.6%，成本最低
  效果：损失 1-2%
  适用：快速原型、资源紧张

r = 16：
  参数 3.2%
  效果：损失 0.1%
  适用：大多数实际应用

r = 64：
  参数 12.8%
  效果：几乎无损 ≈ 99.9%
  适用：追求最优效果

r = 1000：
  等价于 Full Finetune
  参数 100%，成本最高
```

### 1.8 LoRA 的缺点

1. **效果略差**：最多损失 5%，但通常可接受
2. **灵活性受限**：只能改变模型在低维方向的行为
3. **多适配器冲突**：多个 LoRA 同时使用可能互相干扰
4. **秩的上限**：无法改变高维方向上的行为

### 1.9 何时选择 LoRA vs Full Finetune

**选 LoRA**：
- ✓ 资源有限（GPU、显存不足）
- ✓ 需要快速迭代
- ✓ 单一业务微调（充分数据）
- ✓ 需要部署多个版本

**选 Full Finetune**：
- ✓ 完全改变模型行为
- ✓ 有充足预算
- ✓ 需要最优效果
- ✓ 多个业务需要同时工作

### 1.10 QLoRA 补充

**QLoRA = LoRA + 量化**

```
量化的概念：
- Float32：32 位浮点数（精度高，占用大）
- Int8：8 位整数（精度降低，占用 1/4）
- Int4：4 位整数（精度更低，占用 1/8）

效果：
- LoRA 显存节省：100% → 1.6%（参数少）
- 量化显存节省：100% → 25%（精度低）
- 组合效果：显存节省达 40 倍！

实例：
- 7B 模型 Full Finetune：需要 80GB 显存
- 7B 模型 LoRA：需要 30GB 显存
- 7B 模型 QLoRA：需要 16GB 显存（24GB GPU 可跑）
```

## 📖 模块 2：Embedding 和分词原理（进行中）

### 2.1 问题的开始

**核心问题**：文字如何变成模型可处理的数字？

```
过程：
文字 → 分词 → Token ID → Embedding → 向量 → Transformer处理
```

### 2.2 分词的必要性

**为什么需要分词？**

三个选择及问题：

1. **按字处理**
   - 中文：2 万个字，词汇表太大
   - 英文："a", "A", "a." 都算一个字？

2. **按词处理**
   - 问题：英文有多少个词？"learning", "learnings", "learned" 怎么算？
   - 新词无法处理

3. **按子词处理** ✓ （现代 LLM 都用这个）
   - "learning" → ["learn", "ing"]
   - 能处理新词、多语言、代码

### 2.3 什么是 Token？

```
Token = 最小有意义的单位（通常不是完整的词）

例子：
- "hello" → ["hello"]（一个 token）
- "learning" → ["learn", "ing"]（两个 token）
- "Hello123" → ["Hello", "123"]（两个 token）
- "hello\n" → ["hello", "\n"]（两个 token）

Token 大小（文本长度）：
- 英文平均：1 词 = 1.3 token
- 中文平均：1 词 = 1 token
- 代码平均：1 词 = 1.5 token
```

### 2.4 三种分词算法

#### **算法 1：BPE（Byte-Pair Encoding）**

**思想**：从最小单位开始，不断合并频繁出现的相邻对

**过程示例**：
```
初始：["h", "e", "l", "l", "o", "w", "o", "r", "l", "d", ...]

第 1 步：统计相邻对的频率
  ("l", "l")：出现 3 次 ← 最频繁！
  ("e", "l")：出现 2 次

第 2 步：合并最频繁的对
  替换所有 ("l", "l") → "ll"
  结果：["h", "e", "ll", "o", "w", "o", "r", "l", "d", ...]

第 3 步：重新统计
  ("e", "ll")：出现 ? 次 ← 现在最频繁！

第 4 步：合并...

重复数千次，直到词汇表大小 = 目标大小（通常 50K）
```

**特性**：
- 完全基于统计，无需语义知识
- 对任何语言、代码都能工作
- 被 GPT 系列使用

#### **算法 2：WordPiece**

**思想**：从词汇表开始，对未知词进行贪心分割

**过程示例**：
```
预定义词汇表：["hello", "world", "##ed", "##ing", "##ly", ...]
注意：## 表示"非词首的子词"

处理新词："learning"

步骤：
1. "learning" 在词汇表中吗？否
2. 尝试缩短："learn" 在词汇表中吗？是！
3. 剩余："ing"
4. "##ing" 在词汇表中吗？是！

结果："learning" → ["learn", "##ing"]
```

**特性**：
- 需要预定义词汇表
- 效果通常好于 BPE
- 被 BERT、RoBERTa 使用

#### **算法 3：SentencePiece**

**思想**：端到端学习，不依赖预分词

```
输入原始文本（包括空格）
"HelloWorld机器学习"

SentencePiece 学习最优切分点

结果例子：
["▁Hello", "World", "▁", "机器", "学习"]
（▁ 表示空格）

特性：
- 不需要预分词
- 对任何语言一致
- 被 T5、PaLM 使用
```

### 2.5 词汇表大小

```
词汇表 = 所有可能 Token 的集合

GPT-2：
  50,257 个 Token

GPT-3.5：
  ~100,000 个 Token

关键影响：
- Embedding 层大小 = vocab_size × embedding_dim
- GPT-3：100K × 12,288 = 1.2 亿参数（仅 Embedding 层！）
```

### 2.6 Embedding 的本质

**Embedding = 查表过程**

```
构建：Embedding 层是一个 vocab_size × embedding_dim 的矩阵

Token ID 0 → [0.1, 0.2, 0.3, ..., 0.5]（768 维向量）
Token ID 1 → [0.8, 0.1, 0.9, ..., 0.2]
Token ID 2 → [0.5, 0.5, 0.5, ..., 0.5]
...
Token ID 100000 → [0.3, 0.4, 0.2, ..., 0.1]

使用：对输入序列查表

输入 Token IDs：[5, 23, 15, 8]
输出：
  ID 5 → [0.1, 0.2, ..., 0.5]
  ID 23 → [0.8, 0.1, ..., 0.2]
  ID 15 → [0.5, 0.5, ..., 0.5]
  ID 8 → [0.3, 0.4, ..., 0.1]

结果：4×768 矩阵（4 个 token，每个 768 维）
```

### 2.7 Embedding 编码语义

**向量可以表达语义关系**

```
例子：性别关系在向量空间中被编码

v_king = [0.1, 0.2, 0.3, ..., 0.5]
v_queen = [0.1, 0.2, 0.3, ..., 0.4]
v_man = [0.8, 0.1, 0.9, ..., 0.2]
v_woman = [0.8, 0.1, 0.9, ..., 0.1]

关键性质：
v_king - v_queen ≈ v_man - v_woman

含义：
"king 相对 queen"的差异 ≈ "man 相对 woman"的差异
性别关系被向量维度编码了！
```

### 2.8 Embedding 维度的选择

```
常见维度：
- BERT：768
- GPT-2：768
- GPT-3：12,288
- T5-base：768
- T5-large：1,024

规律：
维度越大 → 编码信息越多 → 表达能力越强 → 参数越多 → 成本越高

参数量对比：
GPT-2（50K vocab, 768 dim）
  参数：50K × 768 ≈ 3,840 万

GPT-3（50K vocab, 12,288 dim）
  参数：50K × 12,288 ≈ 6 亿（多 15 倍！）

为什么 GPT-3 要用大维度？
- 需要编码更复杂的语义
- 需要处理更多"概念维度"
- 模型容量大，需要足够的 Embedding 维度配套
```

### 2.9 Embedding 的学习过程

**Embedding 不是随意初始化的，也是通过梯度下降学出来的！**

```
预训练开始：
  Embedding 随机初始化
  v_king = [随机, 随机, ..., 随机]

第 1 步：前向传播
  输入文本 → 分词 → Token IDs
  → 查 Embedding 表获得向量
  → 送入 Transformer 处理
  → 生成输出

第 2 步：计算损失
  预测的下一个词 vs 实际下一个词

第 3 步：反向传播
  计算 dEmbedding/dloss
  告诉我们 Embedding 向量应该怎么改

第 4 步：更新
  v_king_new = v_king_old - α × dEmbedding

第 5 步：重复
  处理下一个数据...

最终：
  经过数十亿次更新
  Embedding 学到了语义信息
  相似词的向量相似，不同词的向量不同
```

### 2.10 完整的文本→向量流程

```
输入："机器学习很有趣"

第 1 步：分词
  选择分词器（BPE、WordPiece 等）
  → ["机", "器", "学", "习", "很", "有", "趣"]

第 2 步：ID 转换
  查词汇表将词映射到 ID
  → [523, 125, 234, 567, 89, 456, 234]

第 3 步：Embedding 查表
  Embedding 层（vocab_size × embedding_dim）
  查表获得向量
  → [
       [0.1, 0.2, ..., 0.5],    # ID 523 的 768 维向量
       [0.8, 0.1, ..., 0.2],    # ID 125 的 768 维向量
       ...
     ]
  结果：7×768 矩阵

第 4 步：位置编码
  加入位置信息（Transformer 无法感知顺序）
  每个向量加上"这是第几个位置"的向量
  → 融合后的向量

第 5 步：Transformer 处理
  经过 96 个 Attention 层 + FFN 层
  每层学到更高层的语义

第 6 步：输出层
  最后一层输出：logits（未归一化的概率）
  softmax → 概率分布
  argmax → 最可能的下一个 Token
```

## 🤔 检查题和答案

### Q1：为什么不能直接用 Token ID 送入 Transformer？

**用户答案**：Token ID 只是数字，只代表词在词汇表中的位置，并不具有实际含义

✅ **评分**：⭐⭐⭐⭐⭐ 完全正确

**深化理解**：
```
Token ID 的问题：
- ID = 5，只表示"词汇表中第 5 个词"
- 不包含任何语义信息
- ID = 5 vs ID = 6，无法看出两个词的关系

为什么不能用 ID 送入 Transformer：
- Transformer 做 Q·K^T 计算，这是向量点积
- 如果输入是 [5, 2, 8, 1] 这样的数字
- 计算 5·2 = 10 毫无意义（这不是语义相似度）
- 无法学到"king 和 dog"之间的关系

解决方案：用 Embedding
- ID 5 → Embedding → [0.1, 0.2, ..., 0.5]（768 维向量）
- 现在 Q·K^T 计算的是向量相似度，有语义意义
```

---

### Q2：如果 Embedding 维度很小（比如 8 维），会怎样？

**用户答案**：参数小，导致词汇表达的意义变少，最终使模型能力下降

✅ **评分**：⭐⭐⭐⭐⭐ 完全正确

**深化理解**：
```
Embedding 维度的作用 = 能表达多少个"独立的语义维度"

如果 embedding_dim = 8（而不是 768）：

后果 1：信息严重损失
  768 维 → 8 维，压缩比 1:96
  就像把 768 像素的图像压成 8 像素
  细节全丢了

后果 2：相似词无法区分
  "King" 和 "Queen" 的差异无法表达
  768 维能精细表达性别、权力等特征
  8 维可能被压成几乎一样的向量

后果 3：Attention 无法学习
  Q·K^T 的结果维度也变小
  模型无法学到复杂的语义关系

后果 4：模型性能严重下降
  embedding_dim = 768：准确率 95%
  embedding_dim = 64：准确率 60%
  embedding_dim = 8：准确率 30%（接近随机）
```

---

### Q3：BPE 和 WordPiece 在实际应用中有什么区别？

**用户答案**：因为二者分割方式不同，BPE 基于统计规律，WordPiece 基于词汇表中的词进行分割，最终会导致分词结果不同，进而导致 token 不同

✅ **评分**：⭐⭐⭐⭐⭐ 完全正确

**深化理解**：

#### BPE 分割示例
```
单词："learning"

BPE 过程（统计驱动）：
第 1 步：["l", "e", "a", "r", "n", "i", "n", "g"]
第 2 步：统计频率
  ("n", "i")：最频繁
  → 合并为 "ni"
  结果：["l", "e", "a", "r", "ni", "n", "g"]
第 3 步：继续统计和合并...
第 N 步：["learn", "ing"]

结果：2 个 token
```

#### WordPiece 分割示例
```
单词："learning"

WordPiece 过程（词表驱动）：
预定义词汇表：["learning", "learn", "##ing", ...]

第 1 步："learning" 在词汇表中吗？是！
结果：["learning"]

结果：1 个 token
```

#### 实际对比
```
同一句话在不同模型中：

句子："The quick brown fox"

GPT 模型（BPE）：4 个 token
BERT 模型（WordPiece）：可能是 5-6 个 token

原因：
- 分割算法不同
- 词汇表内容不同
- 分割策略不同（统计 vs 贪心）

结果：
同样的文本，不同的 token 数，需要不同的位置编码
```

#### 未知词处理
```
单词："neuromorphic"

GPT（BPE）：
  算法能自动分割："neuro" + "morph" + "ic"（3 个 token）

BERT（WordPiece）：
  逐字贪心匹配（开始不在词表，逐渐缩短）
  结果："n" + "euro" + "morph" + "ic"（4 个 token）

BPE 对未知词的处理更优雅
```
